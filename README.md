# go_games_microservice
Игровой Микросервис: Сервер обсчета игровой статистики

Игровые сервера (например, для мультиплеерных шутеров или шахмат) не используют HTTP, они работают на чистом TCP ради скорости. Ваш сервис — это координатор. Он держит TCP-соединения с запущенными игровыми комнатами. Когда матч заканчивается, комната по TCP шлет пакет данных, Go-сервис его парсит, валидирует и сохраняет в Postgres.

Бизнес-цель: Сервер координирует игровые комнаты (матчи). Сами игровые сервера (ноды) держат постоянный TCP-канал для передачи логов матча в реальном времени. В это же время игроки и админы через сайт (REST API) могут смотреть live-статистику текущих игр и профили игроков.

1. Функциональные требования
  - TCP-интерфейс (Для игровых серверов/комнат):
    - Порт: :8085 (Сырой TCP).
    - При старте матча игровая комната подключается по TCP и авторизуется: START_MATCH:match_uuid_999\n.
    - Во время игры комната шлет события (события текстовые, разделены \n):
      - EVENT:player_1:kill:player_2\n
      - EVENT:player_1:score:100\n
    - При завершении матча: END_MATCH\n (после этого сервер закрывает TCP-соединение).
  - REST API (Для сайта/дашборда):
    - Порт: :8080 (HTTP, JSON).
    - GET /api/v1/matches/live — показать список ID всех матчей, которые идут прямо сейчас (активные TCP-сессии).
    - GET /api/v1/matches/{id}/stats — показать live-статистику конкретного матча (сколько фрагов/очков набрано прямо сейчас в памяти сервера).
    - GET /api/v1/players/{nickname}/history — получить историю сыгранных матчей игрока (из Postgres).
2. Стек технологий
  - Язык: Go (только стандартная библиотека net для TCP, для REST можно взять легкий ://github.com или gin).
  - База данных: PostgreSQL (хранение завершенных матчей и общего профиля игроков).
  - Кэш/Память: Redis (для быстрого подсчета live-статистики активного матча и инкремента очков).
  - Инфраструктура: Dockerfile и docker-compose.yml (Go + Postgres + Redis).
3. Критерии приемки и требования к качеству:
  - Thread-Safety (Потокобезопасность): Каждое TCP-подключение работает в своей горутине. Если вы храните список матчей в памяти (map), доступ к этой карте обязан быть защищен с помощью sync.RWMutex, иначе приложение упадет в панику при первом же параллельном запросе из REST API.
  - Разделение слоев (Clean Architecture):
    - Код, который парсит строчку EVENT:player_1:kill..., не должен знать про SQL-запросы в Postgres. Он должен вызвать метод бизнес-логики usecase.RegisterKill(...).
    - HTTP-хэндлеры не читают из сокетов напрямую. Они обращаются к общему менеджеру состояния (Usecase/Service).
    - Graceful Shutdown (Чистое завершение): Если вы нажимаете Ctrl+C, сервер не должен обрывать платежи/матчи посреди записи. Он должен дождаться сохранения данных, закрыть все сокеты, отключиться от БД и только потом завершить процесс.
    - Контексты (context.Context): Все сетевые запросы к Postgres и Redis должны принимать ctx. Если клиент ушел, сервер должен мгновенно отменить тяжелый SQL-запрос, чтобы не тратить ресурсы.
    - Тестируемость: Слой бизнес-логики должен быть покрыт Unit-тестами. Для этого зависимости (база данных, менеджер комнат) должны передаваться через interfaces.
